Terraform File Structure
Purpose

As Terraform projects grow, keeping everything inside a single main.tf becomes difficult to manage.

Terraform file structure helps:

organize configurations
improve readability
separate concerns
simplify collaboration
Important Concept

Terraform automatically reads:

all *.tf files

inside the current directory.

There is NO execution order based on filename.

Terraform internally:

merges all .tf files
builds dependency graph
resolves relationships automatically
Example Beginner-Friendly Structure
project/
├── provider.tf
├── backend.tf
├── input_variables.tf
├── terraform.tfvars
├── tf_az_resource_group.tf
├── tf_az_storage_account.tf
├── outputs.tf
├── backend.sh
└── .gitignore
File Responsibilities
provider.tf

Defines:

Terraform providers
provider versions
provider authentication configuration

Example:

terraform {
  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 4.0"
    }
  }
}

provider "azurerm" {
  features {}
}
backend.tf

Defines:

remote backend configuration
where Terraform state is stored

Example:

terraform {
  backend "azurerm" {
    resource_group_name  = "rg-tf-backend"
    storage_account_name = "tfbackendstorage01"
    container_name       = "tfstate"
    key                  = "terraform.tfstate"
  }
}
input_variables.tf

Defines input variables accepted by the project.

Example:

variable "location" {
  type    = string
  default = "Central India"
}
terraform.tfvars

Stores actual variable values.

Example:

environment = "test"
Resource Files

Used for infrastructure resources.

Examples:

tf_az_resource_group.tf
tf_az_storage_account.tf
outputs.tf

Displays useful deployment outputs.

Example:

output "StorageAccountName" {
  value = azurerm_storage_account.storage01.name
}
.gitignore

Prevents sensitive/unnecessary files from entering Git.

Example:

**/.terraform/*
*.tfstate
*.tfstate.backup
*.tfvars
🧠 Terraform Variables
Purpose

Variables make Terraform code:

reusable
dynamic
cleaner
environment-independent
Input Variables

Defined using:

variable

Example:

variable "environment" {
  description = "Deployment environment"
  type        = string
  default     = "dev"
}
Using Variables

Variables are referenced using:

var.variable_name

Example:

location = var.location
Default Values

default acts as:

fallback value

Terraform uses default only if no other value is supplied.

Variable Files

Terraform automatically loads:

terraform.tfvars

and:

*.auto.tfvars
Important Observation

File:

variables.tfvars

is NOT auto-loaded.

Must use:

terraform plan -var-file="variables.tfvars"
Runtime Variables

Variables can be passed directly during execution.

Example:

terraform apply -var="environment=prod"
Environment Variables

Terraform automatically reads environment variables using:

TF_VAR_

Example:

export TF_VAR_environment="prod"
Variable Precedence

Highest priority wins.

Order:

1. Runtime variable (-var)
2. Variable file (-var-file)
3. terraform.tfvars
4. Environment variable
5. Default value
Example

Variable definition:

default = "dev"

terraform.tfvars:

environment = "test"

Final value becomes:

test

because tfvars overrides default.

🧠 Tags Using Variables

Example:

tags = {
  environment = var.environment
}

If:

environment = "test"

then Azure resource receives tag:

environment = test
🧠 Terraform Locals

Locals are reusable internal values.

Defined using:

locals

Example:

locals {
  common_tags = {
    created_by = "Ivan"
    managed_by = "Terraform"
  }
}

Usage:

tags = local.common_tags
Difference Between Variables & Locals
Variables
external input
configurable by users
Locals
internal reusable values
defined within Terraform configuration
🧠 Output Variables

Outputs display useful resource information.

Example:

output "StorageAccountName" {
  value = azurerm_storage_account.storage01.name
}

After:

terraform apply

Terraform displays:

StorageAccountName = "tfstorageaccount00021211"
🧠 Terraform State

Terraform stores infrastructure information in:

terraform.tfstate

State acts as Terraform’s memory.

It stores:

resource IDs
metadata
dependencies
infrastructure mapping
Why State Is Important

Terraform compares:

Desired State
vs
Current State

using state file.

This enables:

idempotency
updates
dependency tracking
infrastructure reconciliation
Local State Problem

Local state:

not shared
risk of deletion
poor collaboration
not production friendly
Remote Backend

Terraform supports remote backends for centralized state management.

In Azure:

Azure Storage Account
Blob Container

are commonly used.

Remote Backend Architecture
Terraform
   ↓
Azure Storage Account
   ↓
Blob Container
   ↓
terraform.tfstate
Backend Bootstrap Concept

Terraform backend infrastructure is usually:

manually created
OR
created using Azure CLI scripts

Reason:

Terraform cannot store state
inside infrastructure that does not yet exist.
backend.sh Purpose

Used to create:

Resource Group
Storage Account
Blob Container

for remote backend storage.

This infrastructure is NOT managed by Terraform.

Terraform Backend Configuration

Example:

backend "azurerm" {
  resource_group_name  = "rg-tf-backend"
  storage_account_name = "tfbackendstorage01"
  container_name       = "tfstate"
  key                  = "terraform.tfstate"
}
Important Backend Concept
key

represents:

state file name inside blob container

Useful when storing multiple project state files in same backend.

🧠 Terraform State Reconciliation

Terraform does NOT rerun everything blindly.

Terraform:

compares actual infrastructure
with desired configuration

and changes only what is required.

Important Terraform Actions
Create
+
Modify In-Place
~

Example:

LRS → GRS
Destroy
-
Replace Resource
-/+

Occurs when immutable properties change.

Example:

storage account name
certain networking properties
🧠 Terraform Idempotency

Running:

terraform apply

multiple times produces consistent infrastructure state.

Terraform changes only missing or modified resources.

🧠 Terraform Dependency Handling

Terraform automatically detects dependencies using references.

Example:

resource_group_name = azurerm_resource_group.rg.name

Terraform understands:

Storage Account depends on Resource Group

No manual ordering required.
